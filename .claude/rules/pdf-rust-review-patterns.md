# flpdf レビュー頻出パターン（Gemini Code Assist 傾向ルール）

マージ済みPR #246〜#260 の Gemini Code Assist レビュー35件を分析し、
このコードベースで繰り返し指摘される問題を予防ルール化したもの。
コードを書く前・レビューする前に必ず確認すること。

---

## 1. 不要な `.clone()` を作らない（パフォーマンス／メモリ）

**最頻出の指摘**。resolve系APIは多くが所有 `Object` を返すため、
直後の `.clone()` は二重割り当てになる。特に stream を含むツリーの
ディープクローンは OOM リスクになる。

### ルール
- `resolve()` / `resolve_borrowed()` の戻り値を、所有が必要だからといって
  即 `.clone()` しない。`resolve()` は**所有 `Object` を返す**ので直接ムーブして使う。
  借用で十分なら `resolve_borrowed()` のまま借用として扱う（クローンすると借用の意図が無効化される）。
- `Dictionary` から値を取り出すとき、所有が要るなら `get().cloned()` ではなく
  `remove()` でムーブする。
- ツリーを書き換えるときは `&Object` を受け取って全体をディープクローンせず、
  `&mut Object` で in-place に書き換える（stream の raw データ複製を避ける）。
- 条件付きでしか使わないフィールドは、Dictionary 全体をクローンせず
  必要なキー（例: `Kids`）だけ事前にクローンし、本体は `set_object` 等でムーブする。
- debug assertion で「元バイト列全体」を保持しない。長さ + MD5 ハッシュ（既存依存）で
  検証し O(N) 割り当てを避ける。

### 該当例
PR #247（7〜8箇所）, #259, #254, #253, #258, #246

## 2. PDF 間接参照（indirect reference）は必ず解決する（堅牢性）

PDF では任意の値が間接参照で格納されうる。`Object::Name(..)` / `Object::Integer(..)`
を直接マッチすると、参照で格納されたケースを取りこぼし、
署名メタデータ・証明書・フィールド種別の**欠落**につながる。

### ルール
- Dictionary の値を判定する前に `pdf.resolve(..)` で参照を解決する。
  特に以下は間接参照で来うる前提で扱う:
  - 署名フィールドの種別 `/FT`、フィールド名 `/T`
  - 署名 Dict のキー（`/SubFilter`, `/Name`, `/M`, `/Reason`, `/Location`,
    `/ContactInfo`, `/Cert`）
  - `/ByteRange`、配列要素、`/DA` などの継承属性
- 型を見るときは `Object::Integer` への直接マッチではなく
  `Object::as_integer().and_then(..)` のようなアクセサ経由にする（堅牢かつ慣用的）。
- ヘルパー（`name_entry` / `text_entry` / `certificate_entry` 等）は
  `pdf` を受け取り、内部で参照解決してから値を返す設計にする。

### 該当例
PR #252（4件）, #253, #256, #254

## 3. 外部由来の整数は符号なしキャスト前に検証する（オーバーフロー）

PDF から読んだ `/Count` などの値を `as usize` でキャストすると、
負値が巨大値にラップする。32bit ターゲットではプラットフォーム依存バグになる。

### ルール
- 外部入力の整数を `as usize` する前に**非負チェック**する。
  負値や不正値はエラーにするか安全な既定値にフォールバックする。
- ページ数のような累積値は途中まで `i64` で保持し、最後に検証してから変換する。
- 範囲を表す値（`remove.start` / `remove.end` 等）は使用前に
  `start <= end` を検証する。不正範囲のまま走査ロジックに渡さない。

### 該当例
PR #251（`page_splice.rs`、4件）

## 4. オブジェクトグラフ走査は境界と深さを制御する（DoS／正しさ）

PDF のオブジェクトグラフはクロスリンク・循環・深いネストを持ちうる。
無制限に走査すると、意図しない範囲の取り込み、スタックオーバーフロー、
遅延ロード設計の破壊を招く。

### ルール
- ページ単位の閉包・走査では、他の `Page` / `Catalog` Dictionary に到達したら
  そこで**走査を止める**（visited には残すが子へ辿らない）。兄弟ページの内容を取り込まない。
- Pages ツリーを遡らないよう、Dictionary でも stream Dictionary でも
  `/Parent` キーは走査からスキップする。
- 再帰的なツリー走査（`walk_field_tree` 等）には**深さ上限**を設ける
  （例: `DEFAULT_MAX_ACROFORM_DEPTH`）。不正PDFでのスタックオーバーフロー(DoS)を防ぐ。
- 「対象の探索」は全 live object の総当たり（`live_object_refs()` 全走査）に頼らず、
  既知の位置（AcroForm `/Fields`, `/Perms`, `/DSS`, `/Annots` 等）に限定する。
  総当たりは flpdf の遅延ロード設計を台無しにする。

### 該当例
PR #250, #255, #254

## 補足

- Gemini の指摘は medium / high に集中し low はゼロ。実害ベースの指摘が中心。
- 純粋なテスト追加PR（#248, #249, #260）は指摘ゼロになりやすい。
- 上記4カテゴリは「resolve所有権の扱い」「PDF間接参照」「符号付き整数の境界」
  「グラフ走査の境界」という、このコードベース低レベル層の構造的な落とし穴。
