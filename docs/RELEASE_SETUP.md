# Release setup & operations

Releases of `flpdf` and `flpdf-cli` to [crates.io] are automated with
[release-plz] and GitHub Actions (`.github/workflows/release-plz.yml`).
Registry authentication uses crates.io [Trusted Publishing] (OIDC) ‚ÄĒ no
long-lived registry token is stored in the repository.

`flpdf` and `flpdf-cli` share one version via `[workspace.package].version`
(`version.workspace = true`) and always release together under a single
`vX.Y.Z` tag.

## Normal release flow

Day to day, the only manual action is **approving and merging the Release PR**.

1. Merge changes to `main` using [Conventional Commits] (`feat:`, `fix:`,
   `fix(linearize): ‚Ä¶`, etc.). The commit type drives the version bump.
2. The `release-pr` job opens/updates a **Release PR** (branch `release-plz-*`)
   containing:
   - the workspace version bump (both crates + flpdf-cli's `flpdf = { ‚Ä¶,
     version = "x.y.z" }` dependency pin, updated automatically),
   - `Cargo.lock`, and
   - the new `CHANGELOG.md` section (commit-based, generated by git-cliff).
3. Review the Release PR. A maintainer **approves** it ‚ÄĒ this satisfies the
   `release-pr-approval` required status check (gate ‚Ď†). Merge it.
4. Merging fires the `release` job, which **pauses for the `release`
   environment's required-reviewer approval** (gate ‚Ď°). Once approved it
   publishes to crates.io (lib ‚Üí cli), pushes the `vX.Y.Z` tag, and creates the
   GitHub Release with the changelog as its body.

### Version bumps (0.x semantics)

release-plz uses SemVer 0.x rules while the crates are pre-1.0: `feat` **and**
`fix` both bump the patch (`0.1.7 ‚Üí 0.1.8`); only a breaking change
(`feat!:` / `BREAKING CHANGE:`) bumps the minor (`0.1.x ‚Üí 0.2.0`). To force a
specific version, edit the Release PR (or run the manual escape hatch below).

### Changelog scope (known limitation)

The single root `CHANGELOG.md` is owned by the `flpdf` (lib) crate;
`flpdf-cli` has `changelog_update = false` so the changelog never grows a
duplicate per-crate section. Consequence: commits touching **only**
`crates/flpdf-cli/` do not appear in the changelog narrative (they still
trigger the release ‚ÄĒ the shared version bumps and flpdf-cli is republished).
A release cycle with only cli-side commits produces an empty section; add a
manual note to the Release PR if that matters for a given release.

## Approval model ‚ÄĒ two gates

| Gate | What it reviews | Mechanism |
|------|-----------------|-----------|
| ‚Ď† Release PR content | The version bump + changelog are correct | `release-pr-approval` required status check (`release-pr-approval-gate.yml` + main branch ruleset) |
| ‚Ď° Release execution | Authorizes the irreversible crates.io publish + tag | `release` environment required reviewers (also stamps the OIDC subject for Trusted Publishing) |

Gate ‚Ď† is reported by a workflow but **enforced by the branch ruleset**; gate
‚Ď° is enforced by the environment's reviewer setting. Both are one-time repo
settings (below) ‚ÄĒ the workflows alone do not enforce them.

## One-time setup (manual)

Everything below is configured once in GitHub / crates.io; the workflows cannot
create it for you.

### 1. GitHub App (`RELEASE_APP`)

The Release PR must be authored by a GitHub App, not the default
`GITHUB_TOKEN` ‚ÄĒ a PR/commit authored by `GITHUB_TOKEN` does not trigger other
workflows, so CI would never run on the Release PR, and the release tag/GitHub
Release would not be able to fire follow-up workflows.

- Create (or reuse) a GitHub App installed on this repository with repository
  permissions: **Contents: Read and write**, **Pull requests: Read and write**,
  **Workflows: Read and write** (the last is needed when a release branch/tag
  spans a commit that touched `.github/workflows/*`).
- Store its credentials as repository secrets:
  - `RELEASE_APP_ID`
  - `RELEASE_APP_PRIVATE_KEY`

### 2. `release` environment with required reviewers

The `release` job references `environment: release`.

- Create a repository **Environment** named `release`.
- Add the maintainer(s) as **Required reviewers**. This is gate ‚Ď°.

### 3. crates.io Trusted Publisher (per crate)

Trusted Publishing is configured per crate. On each crate's crates.io page ‚Üí
Settings ‚Üí Trusted Publishing ‚Üí add a GitHub publisher:

| Field | Value |
|-------|-------|
| Repository owner | `fulgur-rs` |
| Repository name | `flpdf` |
| Workflow filename | `release-plz.yml` |
| Environment | `release` |

- <https://crates.io/crates/flpdf/settings>
- <https://crates.io/crates/flpdf-cli/settings>

The `Environment` **must** be `release` ‚ÄĒ it must match the environment the
`release` job runs in, or the OIDC exchange is rejected.

### 4. `release-pr-approval` required status check

- In the `main` branch **ruleset**, add **`release-pr-approval`** as a
  **required status check**. This is gate ‚Ď†. Without it the gate workflow writes
  a status that nothing enforces, and a Release PR could be merged unreviewed.
- Non-release PRs receive an immediate `success` for this context, so the check
  never blocks ordinary contributions.

## Manual escape hatch

If a real release wasn't detected from a push (e.g. the Release PR was
squash-merged with a hand-edited subject), trigger **Release-plz** manually:
Actions ‚Üí *Release-plz* ‚Üí *Run workflow* on `main`. The `release` job still
pauses on the `release` environment approval before publishing.

## Deferred: prebuilt binaries

Platform binary attachment (`release:published` ‚Üí build matrix ‚Üí attach ‚Üí
un-draft) is tracked separately (flpdf-ift.7) and is **not** wired up here. When
it lands: set `git_release_draft = true` on the `flpdf` package in
`release-plz.toml`, add a `v*`-tag-triggered `release.yml`, and add a tag
protection ruleset restricting `v*` tag creation to the release App (to close
the direct-tag-push bypass).
