# qpdf compatibility workflow

flpdf aims for writer-level parity with [qpdf](https://github.com/qpdf/qpdf).
This document describes how that parity is measured, recorded, and reviewed.

## Overview

qpdf compatibility is tracked at two layers:

1. **The empirical golden matrix** — for each (fixture, flag) tuple in the
   curated corpus, three comparators record whether flpdf and qpdf agree.
   Lives at `tests/golden/compat-matrix.md` and is regenerated by an
   integration test.
2. **The decisions registry** — `docs/qpdf-compat-decisions.md` consolidates
   every deliberate divergence point so contributors know which gaps are
   intentional vs incidental.

## The golden matrix

The golden matrix (`tests/golden/compat-matrix.md`) is a Markdown table with
one row per (fixture, flag) tuple and one column per comparator strategy.
Each comparator cell contains `match`, `diverge`, or `skip` — matching the
verdicts emitted by `support::ComparatorResult` in the harness.

- **`flpdf-sha`** — a 64-bit FNV-1a fingerprint (16-char lowercase hex) of
  flpdf's output bytes for that fixture × flag. It changes whenever flpdf's
  output changes, so the baseline test catches byte-level drift even when
  every comparator verdict stays at `diverge`. Not a git SHA, not
  cryptographic; FNV-1a is stable across Rust / stdlib upgrades, which is
  what a committed snapshot needs.
- **Updating** — re-run the integration test with `BLESS=1` to regenerate the
  matrix from the current outputs:

  ```bash
  BLESS=1 cargo test -p flpdf-cli --test compat_matrix_baseline
  BLESS=1 cargo test -p flpdf-cli --test compat_baseline_static_id
  ```

  Re-blessing must be a deliberate act. The PR template's `Compat matrix`
  checkbox exists to make that intention explicit.

## Comparator strategies

Three comparators are run for every (fixture, flag) pair. Each records
an independent `match` / `diverge` / `skip` result in the matrix column.

- **byte-equal**: byte-for-byte identical output. Strictest comparator;
  a `match` here implies flpdf and qpdf produced literally the same
  bytes and subsumes the two looser comparators.

- **qpdf-json**: both outputs are normalised through `qpdf --json` and the
  resulting JSON trees are compared. This strips whitespace and ordering
  differences while still checking semantic equivalence of the object graph.
  Useful for outputs where byte identity is not achievable but semantic
  equivalence is the goal.

- **structural**: a higher-level structural diff that compares the logical PDF
  structure (cross-reference table layout, object counts, page tree shape)
  without descending into stream contents. A `match` here means the two
  files are structurally equivalent even if their bytes or JSON differ.

## Divergence categories

When at least one comparator reports `diverge` for a (fixture, flag) row, the
divergence falls into one of these categories. The strictness of the
comparators forms a ladder: byte-equal is strictest, then qpdf-json, then
structural.

Triage rule:

- `skip` means the comparator could not run (e.g. `qpdf` is not on PATH
  for `qpdf-json`). Treat `skip` as "no signal" — it neither matches nor
  diverges. CI requires qpdf so production baselines should not contain
  `skip` rows; local runs that see `skip` should re-run with the missing
  dependency installed before triaging.
- If at least one comparator returns `match`, the category is named after
  the **strictest matching comparator** — i.e. how deep the divergence
  reaches before a comparator stops being able to absorb it.
- If at least one comparator returns `diverge` and every other
  comparator either returns `diverge` or `skip` (no `match`), the row
  is in the **structural** category — the divergence reaches all the
  way down to the object graph. A row whose comparators are *all*
  `skip` carries no signal at all and is not triageable until the
  missing dependencies are installed and the test is re-run; do not
  classify such a row as structural.

- **byte-level** (byte-equal diverges; qpdf-json and structural match):
  same object graph and same `qpdf --json` view, but the raw bytes differ
  — whitespace, token formatting, filter flavour. Resolvable via writer
  tweaks. Tracked under .20.13–.20.17.
- **semantic** (byte-equal and qpdf-json diverge; structural matches):
  the qpdf-json projection differs (e.g. different ordering or extra
  fields in `qpdf --json`) but the flpdf-parsed object graph still
  matches. Often a side effect of qpdf's normalisation choices.
- **structural** (all non-skipped comparators diverge): the underlying
  object graph differs — extra Info entries, different /ID generation,
  different cross-reference layout. Tracked under .20.9–.20.12,
  .20.19–.20.20.
- **decision-deferred**: a registry entry in
  `docs/qpdf-compat-decisions.md` exists for this divergence; current
  behaviour is recorded but the final policy is pending review.
- **divergent**: flpdf deliberately differs from qpdf (rationale captured
  in the registry).

## Adding a new fixture

1. Place the source PDF under `tests/fixtures/compat/`.
2. Either commit it as-is (for simple sources) or derive it via
   `tests/golden/regenerate.sh` (preferred when the fixture is a qpdf
   transformation of an existing one — see the script's Phase 1 for
   examples).
3. Run `BLESS=1 cargo test -p flpdf-cli --test compat_matrix_baseline`
   and `BLESS=1 cargo test -p flpdf-cli --test compat_baseline_static_id`
   to populate the new rows in the baselines.
4. Commit the new fixture, new golden references, and updated baselines
   together.

## Updating references when qpdf upgrades

The committed `tests/golden/references/` are tied to the qpdf version
recorded in `tests/golden/README.md`. When the project's CI image upgrades
qpdf:

1. Bump the version note in `tests/golden/README.md`.
2. Re-run `bash tests/golden/regenerate.sh` — this overwrites all
   references with the new qpdf's outputs.
3. Re-bless the baselines (`BLESS=1 ...`) so the matrix reflects the new
   reference set.
4. Review the diff: if qpdf changed semantics for any fixture × flag, the
   baseline drift will be visible and should be explained in the PR.

## The decisions registry

`docs/qpdf-compat-decisions.md` lists every `needs-review` subtask that
represents a deliberate or pending divergence from qpdf. When you
introduce a new such divergence:

1. Open (or identify) a beads issue with the `needs-review` label.
2. Add an entry to the registry with state = `deferred` (or the final
   decision if known) + a one-line rationale + cross-reference to the
   issue ID.

## CI behaviour

The `.github/workflows/ci.yml` workflow runs the baseline tests on every
PR. The `Compat matrix baselines (Linux)` step surfaces drift as an
explicit, identifiable CI failure — its log shows the expected vs actual
markdown diff so reviewers can pinpoint the regressing tuple.

The PR template's `Compat matrix` checkbox is the human checkpoint:
re-blessing a baseline must be a deliberate act, not an accident.

## Where to dig deeper

- `tests/golden/compat-matrix.md` — current empirical state
- `tests/golden/baseline-static-id.md` — static-id-only baseline
